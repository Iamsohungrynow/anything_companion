## 1. 一句话定义

Yorimi 让虚拟角色从"被观看的内容"变成"会陪你的存在"——一个有性格、有记忆、会用表情达意，并且能真正出现在你桌面上的 AI 角色。

## 2. 我们在解决什么问题

ACGN、VTuber、独立游戏的粉丝，对角色的情感是真实的：他们会为皮肤、语音、限定内容和直播互动付费。但角色本身一直是"死"的——图片视频里的静态形象、直播间里短暂出现的对象、游戏里跑固定剧情的 NPC、聊天 App 里一个没有存在感的头像。它不记得你，不会主动，也不在你身边。

另一边，年轻人在学习和生活里最缺的往往不是工具，而是"开始的那一下"。累了、拖了、不想动的时候，待办软件只会塞给你更多任务，没有人先接住情绪。

Yorimi 把这两件事合在一起，回答同一个问题：

> 怎么让一个虚拟角色既有人设和记忆、能长期陪着你，又能在现实桌面上被真正看见？

## 3. 最大的差异化：它在桌面上"在场"

市面上的 AI 陪伴产品都困在屏幕里。Yorimi 已经做出了 **3D 桌面展示设备 Demo**：角色用真实的表情、动作和状态出现在桌面上，对话时会有反应，陪你学习时状态会跟着变。

这种"在场感"不是锦上添花，而是 Yorimi 陪伴感的根。屏幕里能聊天的产品很多，但桌面上那个看得见、有反应的"它"，才是 Yorimi 区别于所有纯软件竞品的核心壁垒，也是路演、漫展、展会现场最能让观众一眼看懂差异的东西。

## 4. 目标用户

**C 端用户**
二次元 / 乙游用户、VTuber 粉丝、独立游戏玩家、OC（原创角色）爱好者，以及需要学习陪伴和轻量鼓励的年轻人。

**创作者端**
VTuber、画师与 Live2D 创作者、独立游戏团队、原创 IP 方，以及校园 ACG 社群。

## 5. 应用场景

### 5.1 角色情感陪伴（AI+文娱）

用户创建或选择一个角色，设定名字、性格、语气、世界观和外观，角色随后以稳定的人设和用户日常聊天、问候、鼓励。关键区别在于：它记得你说过的话和偏好，相处越久越"是你的"，而不是每次对话都从零开始的机器人。

### 5.2 学习陪伴与行动引导（AI+教育）

Yorimi 不像传统任务工具那样只往用户头上加待办，而是先以角色的方式陪伴回应，再把"启动一个任务"拆成一个小到不会有压力的动作。

典型流程：

1. 用户输入："我明天有考试，但现在很累，开始不了。"
2. 角色先用符合人设的语气温和回应，降低压力。
3. 系统识别状态：低动力、疲惫、难以开始。
4. 角色给出一个极小的行动："先打开笔记，复习一个公式就行。"
5. 用户进入 10 分钟学习陪伴。
6. 结束后做一次 check-in，并更新记忆。

### 5.3 创作者 IP 运营

VTuber、画师、独立游戏团队和原创 IP 创作者，把自己的角色上传成 AI 互动服务：固定人设、世界观边界、语气风格、粉丝称呼都可控。角色由此从"一次性内容"变成能持续与粉丝互动、可运营、可商业化的资产。

### 5.4 桌面在场体验

3D 桌面设备让角色从聊天窗口走到现实桌面：当前实物 Demo 可展示预设表情、动作与状态，强化陪伴感与传播性。Wave 2/3 的目标是让 Web/App 的对话状态实时驱动设备展示。

## 6. 核心业务逻辑

**用户侧主循环**
创建/选择角色 → 角色按设定对话 → 系统记录偏好与互动记忆 → 角色在学习、生活场景里陪伴、提醒、推动小行动 → 越聊越懂你，关系沉淀，用户留下。

**创作者侧循环**
上传角色资产 → 角色上线为可互动服务 → 粉丝订阅与付费 → 创作者拿到收益和运营数据 → 反哺角色迭代，平台获得内容供给。

两条循环互相供给：创作者提供内容，用户提供互动和付费，中间的**记忆系统**是把"一次性体验"变成"长期关系"的引擎。

## 7. 产品需求（可执行）

每条需求都配明确验收标准，可被直接测试。

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| FR-1 | 角色创建与设定 | 用户能在引导流程内完成角色名称、性格、语气、使用场景的设定，并生成一个可对话角色 |
| FR-2 | 符合人设的 AI 对话 | 多轮对话中，角色回复不脱离所设定的人设与世界观边界 |
| FR-3 | 状态识别 | 系统能从用户输入中识别学习、日常陪伴、低动力、任务启动等基础状态 |
| FR-4 | 轻量行动引导 | 在低动力或学习场景下，角色给出一个具体、低压力的小行动，并能陪用户走完一次 10 分钟 / check-in 的完整闭环 |
| FR-5 | 长期记忆（Lite） | 跨会话记住用户偏好与关键信息，并在后续对话中正确调用 |
| FR-6 | 表情 / 状态反馈 | 对话内容触发对应的角色表情或状态切换 |
| FR-7 | 3D 桌面设备展示 | 当前设备能展示预设表情、动作与状态，体现现实桌面存在感；Wave 2/3 目标是接入 Web/App 状态同步 |
| FR-8 | Creator Studio 原型 | 创作者能上传一个角色的设定 / 语气 / 语音 / 皮肤，并预览其可互动效果 |

## 8. Wave 1 当前完成与 Wave 2 MVP 目标

**Wave 1 当前已完成**
产品定位、应用场景、核心业务逻辑、可执行产品需求、评测标准、技术架构说明、未来路线图；同时已有前身项目的 Web runtime 原型、OpenAI / fallback 后端逻辑、基础 companion 体验，以及 3D 桌面展示设备实物 Demo。当前版本用于说明方向、验证核心体验与展示技术可行性，不等同于完整商业产品。

**Wave 2 MVP 目标**
将现有 runtime 收敛为 Yorimi MVP，跑通角色创建与设定、符合人设的 AI 对话、状态识别、10 分钟行动闭环、记忆 Lite、表情/状态反馈、设备展示联动、Creator Studio 基础原型。每一项都要可演示、可测试、可被评测标准验证。

**暂不做**（聚焦核心，避免铺太开）
大规模开放 UGC 角色市场、完整移动端上架、硬件量产、完整支付系统、大型 IP 授权、直播平台深度接入、高复杂度 3D 建模生成。

## 9. 评测标准

Wave 1 验证四件事：场景是否清晰、业务逻辑是否成立、产品需求是否可执行、是否有可量化的验证方式。下面这些不是问卷式的主观打分项，而是 MVP 上线后能直接测、并据此判断"成没成、下一步往哪走"的具体指标。

### 9.1 核心能力

| 评测项 | 测试方式 | 通过标准 |
| --- | --- | --- |
| 角色一致性 | 多轮对话检查回复是否守人设 | 设定相关回复符合率 ≥ 90% |
| 创建可用性 | 新用户无引导独立建角色 | 能完成全流程，无关键卡点 |
| 状态响应准确性 | 输入"我不想动"类低动力场景 | 能正确识别并给出具体行动，而非空泛安慰 |
| 记忆有效性 | 跨会话回访验证偏好/信息 | 关键记忆调用正确，无明显遗忘 |
| 桌面存在感增益 | 对比纯屏幕 vs 桌面设备体验评分 | 设备组陪伴感评分显著高于纯屏幕组 |

### 9.2 用户与创作者验证

| 评测项 | 测试方式 | 目标 |
| --- | --- | --- |
| 首次完成率 | 用户能否建完角色并开始对话 | ≥ 70% 测试用户完成 |
| 重复使用意愿 | 是否愿意第二、三天主动再打开 | 多日复访为正 |
| 创作者接入意愿 | 访谈 VTuber / 画师 / 独立游戏团队 | 跑通上传流程且愿意接入 |
| 付费意愿 | 针对语音/记忆/皮肤/限定互动/桌面体验测意向 | 至少一项存在明确付费意向 |

## 10. 边界、版权与安全

**产品边界**　Yorimi 是 AI 角色陪伴与互动产品，不替代专业心理治疗、医疗诊断或现实人际关系，重点是日常陪伴、学习支持与轻量行动引导。

**版权与 IP**　只接受原创、授权或创作者自有角色；创作者上传素材时需确认使用权；Creator Studio 配套审核、下架与投诉机制。

**数据与记忆**　用户可查看、修改、删除自己的记忆；记忆只用于提升陪伴体验，不滥用于无关商业推荐；对未成年用户施加更严格的内容与时长限制，并避免诱导过度依赖。
