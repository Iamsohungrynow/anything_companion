## 1. 架构总览

Yorimi 不是一个聊天机器人，而是一套支撑"角色设定、持续互动、用户记忆、场景陪伴、桌面展示、创作者运营"的完整 AI 角色系统。整体分六层：

1. **体验层** —— Web / App：角色创建、聊天、学习陪伴、记忆展示、设备展示入口。
2. **AI 角色系统层** —— 角色设定、人格与语气控制、场景识别、对话生成、状态输出。
3. **记忆与上下文层** —— 角色设定、用户偏好、互动摘要、长期陪伴信息。
4. **行动陪伴层** —— 把情绪与学习状态转成轻量行动：10 分钟陪伴、check-in、任务提醒。
5. **设备与在场层** —— 把角色的表情、动作、状态同步到 3D 桌面设备。
6. **计划中的创作者运营层** —— Creator Studio：Wave 2/3 原型目标，用于让创作者上传角色资产，把角色变成可运营的 AI 服务。

记忆层是把这套系统和普通聊天机器人区分开的关键：它让角色从"每次重置"变成"越用越懂你"。

## 3. 用户端与核心流程

用户端是主入口，负责创建角色、互动、体验陪伴流程，并查看记忆与设备展示。

| 页面 | 功能 | 阶段目标 |
| --- | --- | --- |
| 首页 | 展示产品定位与入口 | 让人一眼读懂项目 |
| 角色创建 | 创建 / 选择 AI companion | 验证角色设定流程 |
| 聊天 | 与角色 AI 对话 | 验证互动能力 |
| 10 分钟陪伴 | 引导完成短时行动 | 验证"陪伴到行动"闭环 |
| Check-in | 反馈完成状态 | 收集行动结果 |
| 记忆摘要 | 展示角色记住的偏好与互动 | 验证长期关系 |
| 设备展示 | 展示桌面设备与角色状态 | 体现现实存在感 |

核心体验链路：

```text
进入 → 创建/选择角色 → 设定性格语气场景 → 对话 → 用户表达状态
→ 角色回应 → 场景识别 → 陪伴回应或小行动 → 10 分钟陪伴 → Check-in
→ 更新记忆 → 同步状态到 3D 桌面设备
```

## 4. AI 角色系统

这是 Yorimi 的核心：让角色不只是"会回答"，而是稳定地保持人设、语气、记忆和互动方式。

### 4.1 角色 Profile

每个角色有一个基础 Profile，定义其身份、说话方式与展示形态，也是 Creator Studio 角色上传的基础结构。

```json
{
  "character_id": "yorimi_study_001",
  "name": "Yorimi",
  "role": "Study Companion",
  "personality": ["gentle", "supportive", "focused"],
  "tone": "warm and concise",
  "use_case": ["study", "daily companionship", "micro-action"],
  "visual_skin": "anime_style_companion",
  "device_display_mode": "desktop_presence"
}
```

### 4.2 Persona Prompt

Persona Prompt 控制角色的人格、语气和边界——角色是谁、怎么称呼用户、说话风格、能做什么、不能做什么、用户卡住时如何回应、如何把陪伴转成小行动。

```text
You are Yorimi, a gentle AI companion.
Support the user in study, daily routines, and light emotional companionship.
Listen first, respond warmly, then suggest one small, concrete action.
Do not overwhelm the user with long plans.
Do not claim to be a therapist, doctor, or human.
Keep responses concise, supportive, and character-consistent.
```

核心原则：先陪伴，再帮用户迈出一个小行动，而不是催着完成任务。

### 4.3 场景识别

第一阶段不需要复杂分类模型，用"规则 + LLM 判断"即可。

| 场景 | 用户输入示例 | 系统识别 |
| --- | --- | --- |
| 学习陪伴 | "我明天有考试，但开始不了。" | study + low motivation |
| 日常聊天 | "今天有点无聊。" | daily companionship |
| 任务启动 | "该写作业了，但不想动。" | task initiation |
| 休息提醒 | "学了很久，有点累。" | rest support |
| 创作者互动 | "让这个 VTuber 角色和粉丝打招呼。" | creator interaction |
| 设备状态 | "让角色进入专注模式。" | device state update |

输出示例：

```json
{
  "scenario": "study_support",
  "user_state": ["tired", "stuck", "low_motivation"],
  "recommended_mode": "encourage_mode",
  "next_action_type": "micro_action",
  "device_state": "encouraging"
}
```

### 4.4 回复生成

回复不追求信息量，而要符合角色产品的特点：先回应状态（让用户被理解，而非被命令）→ 再给一个具体、低压力、能立刻开始的小行动 → 全程守住角色语气 → 同时输出状态标签驱动表情与设备。

```json
{
  "reply": "我知道你现在很累，我们不用一下子复习完。先陪你开始 10 分钟，好吗？第一步只要打开 finance notes，找到最重要的三个公式。",
  "mode": "encourage_mode",
  "micro_action": "Open finance notes and review three key formulas.",
  "emotion_state": "gentle_encouragement",
  "device_display": "soft_glow_focus"
}
```

## 5. 记忆系统

记忆系统让角色从一次性聊天工具变成长期陪伴伙伴。它不是存聊天记录，而是理解用户偏好、互动历史和有效的支持方式。

| 记忆类型 | 内容 | 用途 |
| --- | --- | --- |
| Character Memory | 角色设定、性格、世界观、语气 | 保持角色一致 |
| User Memory | 用户偏好、称呼、学习习惯 | 个性化陪伴 |
| Session Memory | 当前会话上下文 | 短期对话连贯 |
| Action Memory | 完成过的小行动、check-in 结果 | 优化后续行动建议 |

```json
{
  "user_id": "user_001",
  "character_id": "yorimi_study_001",
  "preferred_name": "Lin",
  "preferred_tone": "gentle and concise",
  "common_stuck_points": ["exam revision", "starting homework"],
  "effective_support_style": ["10-minute sprint", "small first step"],
  "recent_actions": [
    { "date": "2026-06-24", "scenario": "finance quiz", "micro_action": "review three formulas", "check_in": "partly_done" }
  ],
  "memory_summary": "Often feels stuck before exams; responds well to short, gentle 10-minute study actions."
}
```

更新流程：用户输入 → 角色回应 → 场景识别 → 生成 micro-action → 用户完成/部分完成 → check-in → 生成本次摘要 → 更新 User / Action Memory。

边界：记忆可查看、可修改、可删除，只用于提升陪伴体验，不记录无谓的敏感信息，对未成年用户限制记忆深度（完整安全原则见 `01_项目说明文档.md` 第 10 节）。

## 6. 行动陪伴（Companion-to-Action）

Yorimi 的关键差异化：角色不止聊天，还能帮用户从低动力进入一个小行动。

```text
用户："我卡住了。"
→ 先倾听 → 识别低动力 → 切到 Encourage Mode
→ 给一个小行动 → 用户开始 10 分钟 → check-in → 更新记忆
```

| 行动类型 | 示例 | 场景 |
| --- | --- | --- |
| 10 分钟冲刺 | "先专注 10 分钟。" | 学习 / 工作 |
| 开始第一步 | "先打开笔记。" | 拖延 / 难以开始 |
| Check-in | "今天状态怎么样？" | 日常陪伴 |
| 休息提醒 | "休息 3 分钟再继续。" | 疲惫 |
| 复习提示 | "复习三个关键词。" | 考试 |
| 创作者互动 | "生成一段角色粉丝问候。" | 创作者运营 |

```json
{
  "mode": "encourage_mode",
  "user_state": "low_motivation",
  "micro_action": {
    "title": "10-minute finance review",
    "steps": ["Open your finance notes", "Find three formulas", "Read one example question"],
    "duration": 10
  },
  "check_in_options": ["Done", "Partly done", "I got stuck"]
}
```

## 7. 3D 桌面设备架构

设备不是独立于软件之外的硬件，而是 AI 角色系统的展示与互动载体。物理实测参数（9.7 英寸 IPS 高亮屏、Pepper's Ghost 反射成像、驱动板独立循环播放）见 `02_代码运行说明.md` 第 5 节，本节只讲它在系统里的位置。

设备当前承担四件事：让角色从屏幕进入真实桌面（Presence）、按对话与用户状态切换表情/氛围（状态反馈）、在学习休息等场景提供可视化陪伴（场景陪伴）、在路演漫展形成直观记忆点（线下传播）。

第一阶段设备不承载 AI 计算，只接收来自 Web / App 的角色状态：

```json
{
  "character_id": "yorimi_study_001",
  "device_state": "focus_mode",
  "expression": "encouraging",
  "motion": "gentle_idle",
  "display_text": "Let's start with 10 minutes.",
  "background_effect": "soft_focus"
}
```

软件与设备的目标关系，由 Device Bridge 同步层连接：

```text
AI Character Engine → 输出 reply + emotion_state + device_state
→ Web/App 展示文字与头像 → Device Bridge 发送状态 → 设备展示表情/动作/状态
```

Device Bridge 分两步走：**当前** 先做展示型同步（设备播放预设状态，页面或手动切换，重点验证用户是否感知到更强存在感）；**Wave 2/3** 接入局域网 / 蓝牙 / WebSocket 实时同步，做到语音、动作、灯效联动与多角色皮肤支持。

## 8. 计划中的 Creator Studio

Creator Studio 是 Wave 2/3 的创作者端原型目标，不是当前 Wave 1 runtime 已上线功能。它面向 VTuber、画师、独立游戏团队和原创 IP，目标是把角色变成可互动、可运营、可商业化的 AI 服务。

| 模块 | 功能 |
| --- | --- |
| Character Profile | 角色名称、身份、设定 |
| Lore / Worldview | 世界观、故事、角色关系 |
| Boundary Rules | 角色不能说/不能做什么，防人设偏离与内容风险 |
| Tone Samples | 语气样例、口头禅、粉丝称呼 |
| Visual / Voice Assets | 皮肤、表情、动作、授权语音或 TTS 风格 |
| Fan Interaction | 粉丝问候、生日互动、限定剧情 |
| Analytics | 互动量、热门问题、用户反馈 |

计划接入流程：注册/申请白名单 → 上传设定 → 设世界观边界 → 上传语气与视觉素材 → 生成测试版角色 → 小范围粉丝测试 → 调整人设 → 上线订阅/皮肤/语音包/限定互动 → 看数据。

```json
{
  "creator_id": "creator_001",
  "character_name": "Miyo",
  "creator_type": "VTuber",
  "lore": "A cheerful virtual streamer who loves games and late-night study sessions.",
  "boundaries": ["Stay within the official worldview", "Never pretend to be the real human creator", "No unsafe content"],
  "available_skins": ["default", "study_mode", "festival_limited"],
  "commercial_items": ["monthly subscription", "voice pack", "limited interaction"]
}
```

安全机制：只接原创或授权角色、上传前确认版权、提供边界设置与审核下架、创作者可随时改/关角色、不替代真人创作者、对粉丝互动设健康边界。

## 9. 数据模型

核心表（已在上文给出 Profile 与 Memory 的完整结构，这里列出关系骨架）：

```text
User ──< Conversation >── Character
  │                          │
  └──< Memory >──────────────┘
Character ──< DeviceState
Creator ──< Character (via Creator Studio)
```

```json
// Conversation（会话）
{
  "conversation_id": "conv_001",
  "user_id": "user_001",
  "character_id": "character_001",
  "messages": [
    { "role": "user", "content": "我明天有考试，但开始不了。" },
    { "role": "assistant", "content": "我们先不急，我陪你从 10 分钟开始。" }
  ],
  "created_at": "2026-06-24"
}

// DeviceState（设备状态）
{
  "device_id": "device_001",
  "character_id": "character_001",
  "state": "focus_mode",
  "expression": "encouraging",
  "motion": "idle_gentle",
  "last_updated": "2026-06-24"
}
```

User 表区分 `age_mode`（未成年/成年）以触发不同内容边界；Memory 表带 `memory_type` 区分四类记忆。

## 10. 技术栈

当前阶段优先轻量、快迭代、易扩展。下面区分“当前已实现”和“Wave 2 可扩展方向”，避免把路线图误读为已经全部完成。

### 10.1 当前实现

| 层级 | 选型 | 说明 |
| --- | --- | --- |
| 前端 | Standalone HTML + React CDN runtime | 当前部署文件为 `frontend/static/nextstep-companion.html`，文件名为前身技术原型兼容保留 |
| 源码镜像 | `frontend/companion-experience/` | 分模块保存可维护的 React / TypeScript 源码镜像 |
| 后端 | Node.js local server | `server/index.js` 只负责本地启动，业务处理在 `server/http/runtimeHandlers.js` |
| Serverless | Vercel `.mjs` wrappers | `api/` 中的薄封装复用 shared handlers |
| AI 对话 | OpenAI runtime | `OPENAI_API_KEY` 配置时走 live AI |
| Fallback | Schema-compatible mock | OpenAI 缺失、失败或显式 forced 时降级，不假装 GPT tutoring |
| 数据 | In-memory session store | 用于当前 Demo 的会话与轻量记忆 |
| 语音 | Fish Audio proxy + browser `speechSynthesis` fallback | Fish 缺失不阻塞正常聊天 |
| 角色展示 | 静态 companion assets + 视频状态素材 | 默认 companion 图片与设备状态视频 |
| 部署 | Vercel | 当前线上 Demo |

### 10.2 Wave 2 可扩展方向

| 层级 | 方向 | 说明 |
| --- | --- | --- |
| 前端工程化 | React / Next.js / Vite | 将当前单文件 Demo 收敛成更易维护的 Yorimi MVP |
| UI 系统 | Tailwind / shadcn/ui | 统一组件与视觉语言 |
| 数据持久化 | Supabase / Firebase / SQLite | 用户、角色、记忆与 Creator Studio 数据 |
| 实时能力 | WebSocket / SSE | 流式回复与设备状态推送 |
| 角色展示 | Live2D / Sprite / VRM / Three.js | 更丰富的表情、动作与皮肤 |
| 设备联动 | WebSocket / 蓝牙 / 局域网 | 从展示型同步升级为实时设备联动 |
| 创作者后台 | Creator Studio backend | 角色上传、边界设置、预览、数据看板 |

## 11. 工程验收标准

区别于 `01_项目说明文档.md` 第 9 节的产品评测，这里是工程实现层的验收线。

| 模块 | 验收方式 | 通过标准 |
| --- | --- | --- |
| 角色一致性 | 多轮对话 | 回复守人设 |
| 场景识别 | 输入学习/日常/创作者等场景 | 正确识别主要意图 |
| 行动建议 | 输入低动力场景 | 给出具体、可执行的小行动 |
| 记忆写入/读取 | 互动后看摘要、后续对话引用 | 能记录并正确调用 |
| 记忆可控 | 删除/修改记忆 | 用户拥有控制权 |
| 设备状态同步 | 对话后切换设备状态 | 设备展示对应状态 |
| 创作者接入 | 上传角色设定与语气样例 | 生成可测试角色且不偏离设定 |

## 12. 总结

Yorimi 的架构围绕一个目标：让虚拟角色从"可观看的内容"升级为"可互动、可记忆、可陪伴、可展示、可运营"的 AI 角色存在产品。它由四块组成——AI 角色系统（稳定的设定与互动）、记忆系统（长期理解用户）、3D 桌面设备（现实存在感）、计划中的 Creator Studio（让创作者运营自己的 AI 角色）。Wave 1 讲清场景、逻辑、架构与评测标准，后续推进可运行 MVP、创作者试点与设备体验升级（路线见 `04_未来规划.md`）。
